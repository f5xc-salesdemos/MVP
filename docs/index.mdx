# xcsh Product Plan

**Date:** 2026-04-20
**Status:** Draft for executive review
**Audience:** Engineering leadership

## Purpose

This document presents the research, design, and delivery plan for **xcsh**, an agentic shell tool purpose-built for the field sales engineering organization. xcsh is an internal fork of the open-source pi-mono project, extended with an F5 Distributed Cloud domain layer.

The plan follows a standard engineering lifecycle. Each stage is summarized in three bullets to support a time-boxed executive briefing. Implementation-level detail is intentionally omitted; it lives in downstream design documents.

## Naming

The product name is **xcsh**, always lowercase. It is never written as Xcsh, XCSh, or XCSH.

## Plan

### 1. Problem / Opportunity Definition

- Sales engineers spend disproportionate time on information gathering and artifact production, displacing MEDDPICC qualification and close activities.
- Workflows are artisanal: every team member reinvents their own tooling, producing inconsistent quality and non-transferable institutional knowledge.
- The opportunity is to convert repetitive subject-matter-expert work into a shared, governed capability that scales across the field organization.

### 2. Feasibility Study

- Surveyed the commercial and open-source agent landscape; found mature tooling optimized for software development, not field sales engineering.
- Confirmed that a sales-engineering-specific agent is viable by building on an extensible open-source foundation rather than customizing a general-purpose product.
- Established a make-vs-buy position: buy the agent runtime, build the F5 Distributed Cloud domain layer.

### 3. Requirements Definition

- First-class understanding of the F5 Distributed Cloud platform — its API surface, its configuration model, and its operational semantics.
- Curated domain knowledge covering product, competitive, marketing, and enablement material, kept current through a defined ownership model.
- Multi-tenant, secret-aware authentication and automation that meets enterprise security expectations from day one.

### 4. Architecture Design

- Layered architecture: a stable runtime foundation, a clearly separated F5 Distributed Cloud domain layer, and a persona layer that expresses sales-engineering workflows.
- Extension points are explicit and documented, so capability growth does not require changes to the core.
- Separation of concerns between knowledge, automation, and presentation keeps each layer independently testable and maintainable.

### 5. Detailed Design

- Capability decomposition follows the actual sales-engineering workflow: qualify, prepare, demo, present, answer, follow up.
- Each capability is a bounded, composable unit with a defined interface, data contract, and owner.
- State, session, and settings handling are designed around auditability and reproducibility rather than convenience.

### 6. Implementation

- Test-driven development, conventional commit history, and branch-per-change workflow are required, not optional.
- Code quality is gated by automated linting, type checking, and test baselines; no change merges that regresses the baseline.
- Performance-critical paths are identified early and isolated so they can be optimized without destabilizing the broader system.

### 7. Testing

- Every test defends a named, externally observable contract — no placeholder or tautological tests.
- Coverage is layered: unit tests for bounded logic, integration tests for system seams, end-to-end tests for user-visible workflows.
- A measured regression baseline travels with the codebase, so quality trend is observable rather than anecdotal.

### 8. Validation

- Structured field pilot with a defined sales engineering cohort, written success criteria, and measured before/after metrics on qualification and preparation cycle time.
- Knowledge accuracy is audited against a curated ground-truth set maintained by subject matter experts.
- Independent security review of authentication, secret handling, and multi-tenant isolation prior to general availability.

### 9. Launch

- Phased rollout: internal sales engineering cohort, then broader field organization, then partner enablement.
- Release governance through automated versioning, release branches, and traceable change logs — no ad hoc publication.
- Onboarding, enablement, and support ownership are defined before launch, not after.

### 10. Operational Maintenance

- Observability is built in: usage telemetry, structured logs, and health signals surfaced to the operating team.
- Defined cadence for knowledge refresh, dependency updates, and upstream synchronization with the open-source foundation.
- Security posture maintained through routine secret rotation, patch windows, and documented incident response.

### 11. Continuous Improvement & Roadmap

- Field feedback loop is formalized: inputs from the sales engineering organization directly shape the backlog and prioritization.
- Quarterly capability review benchmarks the product against evolving competitor agent platforms.
- Roadmap horizon balances depth — better fidelity on existing tasks — with breadth — new sales-engineering workflows.
